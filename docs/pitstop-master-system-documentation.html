3. Authentication & Roles

المصادقة والأدوار

Roles (public.user_role)

RoleLoginshop_idbranch_idPurpose
customerYes——Books services, favorites, notifications
ownerYesRequired—Owns brand, all branches, creates managers
branch_managerYesRequiredRequiredRuns one branch day-to-day
pending_ownerYesRequired—New merchant awaiting admin approval; is_active=false
adminYes——Super Admin → /admin panel

Employees are NOT auth users — stored in branch_employees only.

Shop visibility: shops.is_active=false hides shop from customer catalog until admin approves.

Customer auth (CustomerAuthContext.tsx)

  • Supabase email/password: signIn, signUp, resetPassword
  • Guest mode: local flag, no Supabase session
  • Blocks shop staff emails from customer login
  • Requires email confirmed for login
  • Storage: @pitstop/customer-session, @pitstop/guest-session

Shop auth (ShopAuthContext.tsx)

  • Resolves from public.users (owner | branch_manager | admin | pending_owner) or legacy shops.owner_email
  • Flags: isOwner, isBranchManager, isAdmin, isPendingOwner
  • registerOwner() — signUp + RPC register_shop_owner
  • Login results: ok | pending_approval | invalid_credentials | …
  • Storage: @pitstop/shop-session (staff email)
  • Branch manager invite uses staffInviteLock to prevent session hijack

Auth storage on web

lib/supabase/authStorage.ts uses localStorage on web for reliable Supabase session persistence.

Role hierarchy (updated)

Super Admin (platform) ├── Approves / rejects pending_owner registrations ├── Views platform stats (bookings, 12% fee) └── Sees all merchants + branches Owner (approved) ├── Creates branches ├── Creates / links Branch Managers └── Sees all branches + reports pending_owner (awaiting approval) └── Shop hidden from customers until admin accepts Branch Manager (one branch) ├── Manages daily operations └── Adds employees (no login) Employees → roster only (branch_employees)

6. Booking Flow

مسار الحجز من البداية للنهاية

End-to-end workflow

Home / Nearby / Favorites → Shop profile (/shop-profile/[shopId]) → [Guest?] → /auth-required → /welcome login → Book (/book/[shopId]?serviceId=...) → Select date + time slot + car + phone + notes → createBooking() → status: pending → Owner notification (+ optional push) → Owner: confirm → confirmed | decline → cancelled → Customer notification + reminders (60 min, 30 min before) → Owner: in_progress → done (or no_show)

Booking statuses

pending → confirmed → in_progress → done | cancelled | no_show

Pricing (lib/booking/storage.ts)

Shop typeDefault price (EGP)
maintenance650
wash220
winch500
other420

Platform fee: 12% of service price (shown in owner PDF reports).

Time slots

Built by buildSlotsForShopDate() from owner weekly hours + existing bookings. Submit disabled until a slot is selected.

Persistence

  • Primary: Supabase bookings table (when configured)
  • Cache: AsyncStorage @pitstop/bookings/v1
  • On Supabase insert failure → error shown to customer (RLS fix: step4 SQL)

23. Super Admin — Overview & Roles

لوحة Super Admin — نظرة عامة

PitStop 2.0 adds a merchant approval gate: new shop owners cannot go live until a platform admin approves them. Existing seed owners (e.g. wash@demo.com) remain pre-approved.

State machine — merchant account

Stateusers.roleusers.is_activeshops.is_activeCustomer catalogApp route
New registrationpending_ownerfalsefalseHidden/welcome (pending message)
ApprovedownertruetrueVisible/shop
Rejectedcustomertruedeleted—/ (customer area)

Admin capabilities

  • View platform-wide booking stats and 12% platform fee estimate
  • List pending merchant registration requests
  • Accept → atomic RPC activates user + shop
  • Reject → deletes shop row, resets user to customer
  • List active merchants with branch counts

26. Merchant Registration Module

وحدة تسجيل التاجر الجديد

UI: app/welcome.tsx — Owner tab

Toggle: Login ↔ Register as merchant (owner_register_link)

Register fields:

  1. Business name (shopName)
  2. Owner full name (name)
  3. Egypt mobile phone — validated
  4. Business address
  5. Shop type chips: wash, maintenance, parts, accessories
  6. Area chips: from catalog areas (Maadi, Nasr City, …)
  7. Email + password (strong password rules)

Submit: registerOwner() in ShopAuthContext

Backend: lib/shop/ownerRegistrationRepository.ts

  1. supabase.auth.signUp with metadata (name, phone, account_type=shop_owner)
  2. supabase.rpc('register_shop_owner', { p_shop_name, p_shop_type, p_area_id, … })
  3. On RPC failure → signOut and return invalid
  4. On success → session restored as pending_owner

Login while pending

Credentials valid but login() returns pending_approval.

AppBootstrap keeps user on /welcome with pending message — no access to /shop or customer booking as owner.
