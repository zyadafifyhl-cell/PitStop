3. Authentication & Roles

المصادقة والأدوار

Roles (public.user_role)

RoleLoginshop_idbranch_idPurpose
customerYes——Books services, favorites, notifications
ownerYesRequired—Owns brand, all branches, creates managers
branch_managerYesRequiredRequiredRuns one branch day-to-day
pending_ownerYesRequired—New merchant awaiting admin approval; is_active=false
adminYes——Super Admin → /admin panel

Employees are NOT auth users — stored in branch_employees only.

Shop visibility: shops.is_active=false hides shop from customer catalog until admin approves.

Customer auth (CustomerAuthContext.tsx)

  • Supabase email/password: signIn, signUp, resetPassword
  • Guest mode: local flag, no Supabase session
  • Blocks shop staff emails from customer login
  • Requires email confirmed for login
  • Storage: @pitstop/customer-session, @pitstop/guest-session

Shop auth (ShopAuthContext.tsx)

  • Resolves from public.users (owner | branch_manager | admin | pending_owner) or legacy shops.owner_email
  • Flags: isOwner, isBranchManager, isAdmin, isPendingOwner, isPremium
  • registerOwner() — signUp + RPC register_shop_owner → signOut (user sees success alert, logs in later when approved)
  • Login results: ok | ok_admin | pending_approval | invalid_credentials | email_not_confirmed | email_login_disabled | …
  • Storage: @pitstop/shop-session (staff email)
  • Branch manager invite uses staffInviteLock to prevent session hijack
  • Web login fix: lib/auth/authMutationLock.ts blocks onAuthStateChange side effects during signIn/signUp to prevent Supabase auth deadlocks (infinite “Signing in…”)

Customer auth — web deadlock fix

Both CustomerAuthContext and ShopAuthContext defer Supabase queries out of onAuthStateChange using setTimeout(0) and skip listeners while beginAuthMutation() is active. Welcome screen shows contextual button labels: Signing in… / Creating account… / Redirecting…

إصلاح تعليق تسجيل الدخول على الويب: لا تُنفَّذ استعلامات Supabase داخل onAuthStateChange أثناء signIn.

Auth storage on web (per-tab sessions)

Web uses tab-isolated storage so you can test owner + customer in parallel browser tabs:

  • lib/supabase/authStorage.ts — Supabase JWT in sessionStorage (not localStorage)
  • lib/storage/webTabAuthStorage.ts — @pitstop/customer-session, @pitstop/guest-session, @pitstop/shop-session in sessionStorage
  • lib/supabase/webTabAuthIsolation.ts — disables Supabase BroadcastChannel cross-tab sync; signOut({ scope: 'local' }) on web so logout in one tab does not revoke the other tab’s session
  • Legacy Supabase keys in localStorage are cleared on first client init

على الويب: كل تاب له session مستقل — owner في تاب و customer في تاب تاني من غير ما يتزامنوا.

Role hierarchy (updated)

Super Admin (platform) ├── Approves / rejects pending_owner registrations ├── Views platform stats (bookings, 12% fee) └── Sees all merchants + branches Owner (approved) ├── Creates branches ├── Creates / links Branch Managers └── Sees all branches + reports pending_owner (awaiting approval) └── Shop hidden from customers until admin accepts Branch Manager (one branch) ├── Manages daily operations └── Adds employees (no login) Employees → roster only (branch_employees)

6. Booking Flow

مسار الحجز من البداية للنهاية

End-to-end workflow

Home / Nearby / Favorites → Shop profile (/shop-profile/[shopId]) → [Guest?] → /auth-required → /welcome login → Book (/book/[shopId]?serviceId=...) → Select date + time slot + car + phone + notes → createBooking() → status: pending → Owner notification (+ optional push) → Owner: confirm → confirmed | decline → cancelled → Customer notification + reminders (60 min, 30 min before) → Owner: Complete → done (or no_show) → Customer: optional 5-star rating on My Bookings (one per shop)

Booking statuses (UI)

pending → confirmed → done | cancelled | no_show

There is no separate “In progress” step in the customer UI or owner wash buttons. Legacy rows with in_progress in the database still display as Confirmed and map to step 2 of the 3-step timeline.

Pricing (lib/booking/storage.ts)

Shop typeBooking price source
washSelected service priceEgp from branch menu; fallback = first active service (no owner “base price” UI)
maintenanceOwner default service price in shop extras, or type default 650 EGP
winchType default 500 EGP
otherType default 420 EGP

Platform fee: 12% of service price (shown in owner PDF reports).

Time slots

Built by buildSlotsForShopDate() from owner weekly hours + existing bookings. Submit disabled until a slot is selected.

Persistence

  • Primary: Supabase bookings table (when configured)
  • Cache: AsyncStorage @pitstop/bookings/v1
  • On Supabase insert failure → error shown to customer (RLS fix: step4 SQL)

Purpose: Reward repeat wash customers. Each completed wash booking adds one stamp; after 5 stamps the customer receives a 100% off coupon valid 90 days.

  • Customer key: id:{customerId} or fallback phone:{phone}
  • Guest users: no loyalty card (login required)
